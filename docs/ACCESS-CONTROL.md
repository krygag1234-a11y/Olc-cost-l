# Контроль доступа к подписке по устройству (allowlist hwid)

## Проблема со старой «рандомизацией»

Рандомизация пути подписки (случайный client-id/путь) **не даёт реальной защиты**:
путь можно узнать/поменять вручную, и любой, кто его знает, получит подписку и
доступ к инстансам. Нужен настоящий контроль — по идентификатору устройства.

## Что реально шлёт olcbox (результат исследования)

Изучен код [olcbox](https://github.com/alananisimov/olcbox)
(`sharedUI/.../data/datasource/LocationsDatasource.kt`,
`data/identity/DeviceIdentityProvider.kt`):

- При запросе подписки в режиме `Identity` (по умолчанию) olcbox отправляет HTTP-GET
  с заголовками:
  - **`x-hwid: install-<32 hex>`** — стабильный per-install идентификатор устройства
    (16 случайных байт, генерируется один раз и сохраняется на устройстве);
  - **`User-Agent: olcbox/<версия>`**.
- Есть fallback `Compatibility` (без `x-hwid`, с «браузерным» User-Agent) — на случай,
  если сервер не понял Identity-запрос.

Значит **сервер (VPS), где лежит подписка, ВИДИТ**: `x-hwid`, `User-Agent` и IP
запрашивающего (`r.RemoteAddr` / `X-Forwarded-For`). Этого достаточно для настоящего
белого списка.

## Что реализовано

Шлюз в обработчике подписки (`subscriptionHandler`) + отдельные файлы состояния:

- **`/var/lib/olcrtc/access-control.json`** — `{enabled, mode, allowed_hwids, allowed_ips}`.
- **`/var/lib/olcrtc/access-attempts.json`** — кольцевой журнал последних 200 попыток
  (`ts, hwid, ip, ua, client_id, path, allowed`), с дедупом повторов автообновления.

Режимы (`mode`):
- **monitor** — пускать всех, но фиксировать попытки в журнал (для сбора hwid).
- **enforce** — неизвестное устройство получает `404` (существование пути не
  раскрывается), известное — подписку.

Если `enabled=false` — контроль полностью выключен (поведение как раньше).

### Почему это покрывает и «кто может подключиться к инстансу»

Чтобы подключиться к инстансу, olcbox берёт `room_id`/`key` **из подписки**. Нет
подписки → нет реквизитов → нет подключения. Поэтому шлюз на уровне подписки
практически ограничивает и подключение. Полноценный enforcement на самом
WebRTC-уровне (по peer-хешу в логах olcrtc-core) потребовал бы патча
pinned-upstream `olcrtc` — вынесено в возможные доработки.

## UI

Модалка **Настройки** → секция **«Контроль доступа (по устройству)»**:
- переключатель «Включить»; режим monitor/enforce;
- список разрешённых hwid (добавить вручную / удалить);
- **журнал попыток** с кнопкой **«Разрешить»** у неизвестных устройств (в один клик
  добавляет hwid в allowlist).

## API (adminAuth)

- `GET /api/access/settings` → `{enabled,mode,allowed_hwids,allowed_ips,updated_at}`
- `PUT /api/access/settings` (тело — те же поля) → сохранить
- `GET /api/access/attempts` → `{attempts:[…]}`
- `POST /api/access/attempts/clear` → очистить журнал
- `POST /api/access/allow` `{hwid?|ip?}` → добавить в allowlist
- `POST /api/access/remove` `{hwid?|ip?}` → убрать

Пример проверки (эмуляция olcbox):
```bash
# monitor: попытка фиксируется, подписка отдаётся
curl -s -H 'x-hwid: install-testdevice' -H 'User-Agent: olcbox/1.0' \
  http://127.0.0.1:8888/<sub-path>/<client-id>
curl -s -u admin:ПАРОЛЬ http://127.0.0.1:8888/api/access/attempts
```

## Взаимодействие с рандомизацией (важно)

Разрешённое устройство может запрашивать подписку по **оригинальному** client-id
(например `bs`) **даже при включённой рандомизации** — для остальных оригинальный
путь под рандомизацией отдаёт 404, доступен только рандомизированный id. Реализовано
через `olcResolveClientIDWithAccess(requestedID, cfg, bypass)`: `bypass = enabled &&
устройство в allowlist`. Попытки фиксируются в журнал **до** резолва, поэтому запросы
логируются и при включённой рандомизации (раньше — нет, т.к. resolveClientID отдавал
404 раньше шлюза).

## Журнал попыток

- Группировка по `(hwid, client_id)`: повторные запросы одного устройства НЕ спамят
  журнал, а увеличивают `count` («×N») и обновляют время последней попытки.
- В UI журнал — в стиле логов: ограниченный по высоте прокручиваемый контейнер;
  follow-newest со скроллбэком (листаешь вверх — пауза; вернулся вниз — через ~1.5с
  снова следует за новыми); адаптация под «автологи» (вкл → авто-обновление и зелёная
  надпись «● автологи» вместо кнопки; выкл → кнопка «Обновить»); кнопка «Очистить».



`access-control.json` включён в бэкап (`backupExtraFiles`), поэтому allowlist
переносится на новый VPS вместе с остальными данными. Журнал попыток — транзиентный,
в бэкап не входит.

## Per-client (индивидуально для подписки)

Кнопка-шестерёнка **⚙** рядом с кнопкой рандомизации (🎲) на карточке клиента
открывает модалку «Доступ к подписке: <id>»: режим (наследовать/выкл/наблюдение/
блокировать), белый список и бан устройств только для этой подписки, журнал попыток
по клиенту с кнопками «Разрешить»/«Бан». Хранится в `access-control.json` →
`clients: {"<id>": {mode, allow[], ban[]}}`. Решение — `olcAccessDecision(ac,
clientID, hwid, ip)`: per-client `mode` переопределяет глобальный (`off` — выключить),
`ban` — жёсткий блок, `allow` — плюс к глобальным `devices`. API `GET/POST /api/access/client`.

### Выборочный контроль ПОДКЛЮЧЕНИЯ per-client
Та же ⚙-модалка, карточка «🔌 Контроль подключений»: `conn_enforce` (bool) +
`conn_scope` (`all`|`selective`) + `conn_instances` (room_id при selective). Действует
на УРОВНЕ ПОДКЛЮЧЕНИЯ (AuthHook olcrtc-core) и **ТОЛЬКО когда глобальный
`enforce_connections` ВЫКЛЮЧЕН** (глобальный приоритетнее). При включённом глобальном
в UI — плашка «выключите глобальный»; per-client настройки сохраняются в JSON и
возвращаются при выключении глобального. `scope=selective` — энфорс только на
выбранных инстансах (room_id), остальные открыты. Хук знает клиента/инстанс из env
`OLCRTC_CLIENT_ID`/`OLCRTC_ROOM_ID` (прокинуты менеджером в процесс olcrtc).

### Мгновенный разрыв при отзыве доступа
При любом сохранении `access-control.json` (`olcAccessSave`) вызывается
`olcDropForbiddenSessions`: по per-instance лог-буферам находит подключённые
устройства и, если устройство больше НЕ разрешено при активном энфорсе подключения
(глобальном или per-client), **перезапускает именно тот инстанс** — активная сессия
нарушителя рвётся сразу (а не только при реконнекте); разрешённые переподключаются.
Без активного энфорса подключения ничего не рвётся.

### Разрешённые IP
`olcbox` шлёт IP при HTTP-запросе подписки — он фиксируется в журнале попыток
(`access-attempts.json`, поле `ip`). В журнале (глобальном и per-client) есть кнопка
**«+IP»** — разрешить этот IP в один клик (`allowed_ips`, backend энфорсит на
подписке). На уровне WebRTC-подключения IP пира недоступен (пиры идут через комнату
по participant-id, не по IP), поэтому IP-allowlist работает по IP запроса подписки.

## ⛔ ПРАВИЛО ДЛЯ РАЗРАБОТЧИКОВ

При изменении формата `access-control.json` или логики шлюза:
1. Обновите `backupExtraFiles()` (если добавили файлы), чтобы контроль доступа
   переносился бэкапом.
2. Синхронизируйте UI-секцию (`patch-olcrtc-manager-panel-access-control-ui.sh`),
   per-client модалку (`patch-olcrtc-manager-panel-client-access-ui.sh`) и
   API-обработчики (`patch-olcrtc-manager-access-control-api.sh`).
3. **Если меняете поля, влияющие на enforcement подключения — синхронизируйте
   структуры в `patch-olcrtc-core-access-hook.sh`** (`olcAccControl`/`olcAccDevice`):
   это ОТДЕЛЬНАЯ копия структуры в olcrtc-core, читающая тот же JSON. Держите
   safe-by-default: fail-open при ошибке и пустом allowlist.
4. Помните про Identity/Compatibility: если хотите заблокировать даже
   Compatibility-fallback (без hwid) — `ban_no_hwid` режет такие запросы (и на
   подписке, и на подключении).

## Контроль на уровне ПОДКЛЮЧЕНИЯ к инстансам (исследование + план)

**Находка (openlibrecommunity/olcrtc):** при подключении сервер логирует
`peer session <sid> opened (peer=<p> device=install-<hex>)` — `device` берётся из
handshake `Hello.DeviceID` (`internal/handshake/handshake.go`, `internal/server/server.go`).
Это **тот же** `install-<hex>`, что olcbox шлёт как `x-hwid` при запросе подписки
(persistent per-install id). Итог: **один allowlist покрывает и подписку, и
подключение**. Слепок НЕ меняется при обновлении приложения, меняется при
переустановке/очистке данных.

**Реализовано (монитор с привязкой):** `GET /api/access/connections` идёт по
**per-instance лог-буферам** (`panelSupervisor.processes`) — менеджер знает
клиента и локацию каждого процесса olcrtc, поэтому `device=install-…` из строк
peer session сопоставляется с подпиской (`client_id`) и инстансом
(`location_name`/`room_id`/`transport`). Фолбэк на journal, если буферы пусты. В
UI — карточка «🔌 Подключения к инстансам» с колонками подписка/инстанс и
кнопками «Разрешить»/«Бан».

**Enforcement на подключении — РЕАЛИЗОВАНО (safe-by-default), требует теста:**
`patch-olcrtc-core-access-hook.sh` ставит `Config.AuthHook` в olcrtc-core
(`internal/app/session`), который на CLIENT_HELLO читает `access-control.json` и
отклоняет неразрешённые/забаненные устройства. **Включается ТОЛЬКО** при
`enabled && enforce_connections` (отдельный флаг, по умолчанию OFF). Любая ошибка
(нет файла/парс) или пустой allowlist → **fail-open** (пускаем) — гарантия, что
хук НЕ рвёт туннели. Глобальный `ban` и `ban_no_hwid` учитываются.
⚠️ РИСК: неверный хук рвёт ВСЕ туннели → перед тем как полагаться, включите
`enforce_connections` и проверьте на РЕАЛЬНОМ устройстве (своё → 200/подключается,
чужое → reject). Тумблер — в UI, «Уровни защиты» → «Доступ к подключению».

**План структуры (глобально + per-client):** глобальный контроль (есть) — ко всем
подпискам/инстансам или выборочно; per-client — кнопка-шестерёнка у кнопки
рандомизации (🎲) на карточке клиента → модалка доступа для ЭТОЙ подписки
(белый список / бан устройств, ограничение подключения ко всем/выборочным инстансам).

## Whitelist по IP (реализовано)

Секция «Контроль доступа» → блок **«Разрешённые IP»**: список + добавить/удалить.
Backend энфорсит `allowed_ips` в `olcAccessDecision` → `olcAccessAllowed` (проверяются
и devices по hwid, и allowed_ips по адресу). Используется для доступа к подписке
**без приложения/hwid** (свой сервер, скрипт). Управление — через те же
`/api/access/{allow,remove}` с телом `{ip}`. `allowed_ips` входит в бэкап
(`access-control.json`).
