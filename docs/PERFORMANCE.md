# Скорость: Tor, direct, WebRTC

## Где потолок скорости

```text
Olcbox (link=tor) → VPS olcrtc → SOCKS Tor :9050 → мост → сеть Tor → exit → сайт
```

| Участок | Типичный предел | Увеличение vCPU/RAM на VPS |
|---------|-----------------|---------------------------|
| Мост (webtunnel/obfs4) | ~5–15 Mbit/s | почти не помогает |
| Exit-релей Tor | ещё один hop | не контролируем |
| Direct (`link=direct`, `*.ru`, CDN) | канал VPS ↔ провайдер | немного при перегрузе CPU |
| Jitsi/WebRTC media | UDP клиент↔VPS | кодирование на клиенте |

**Выше Tor на этом стеке** без смены архитектуры (другой exit не поможет — вы уже *клиент* Tor) **не выжать**. Имеет смысл:

- свежий пул мостов (`olcrtc-tor-bridge-pool.timer`, `fetch-bridge-extra-sources.sh`);
- **webtunnel первыми** в `bridges.conf` (obfs4 — запас);
- `tor-bridge-deep-check.sh` — приоритет `bootstrap_ok`;
- не рестартить Tor без нужды (`healthcheck.sh` → `/admin`).

Подробнее: [TOR-BRIDGES.md](TOR-BRIDGES.md).

## Почему speedtest / curl и Olcbox могут отличаться

| Инструмент | Протокол | Что меряет |
|------------|----------|------------|
| `curl` через SOCKS | один (или мало) TCP-потоков | один Tor-circuit, ощущается «узко» |
| 2ip / speedtest в браузере | часто **много параллельных HTTP** | сумма потоков, визуально выше |
| Olcbox звонок | **WebRTC** (UDP, несколько каналов) | не тот же путь, что один `curl` |

Отсюда фраза: «на curl меньше, WebRTC даёт больше за счёт параллельных потоков» — про **разный трафик**, не про магию VPS.

## Можно ли «запустить много потоков» и ускорить Tor?

**Частично, с оговорками.**

| Действие | Эффект |
|----------|--------|
| Несколько `curl` параллельно через **тот же** SOCKS | часто **один circuit** → прирост 0–30%, не ×N |
| Speedtest с 8+ соединениями | может показать **суммарно** больше Mbit/s, если мост и exit тянут |
| Поднять «потоки» на **сервере** olcrtc | **не ускорит** Tor-путь клиента к зарубежному сайту |
| Больше мостов в `bridges.conf` (12–18) | **запас** при отказе, не суммарная полоса |
| `link=direct` для тестов без Tor | максимум канала VPS (~10+ Mbit/s class) |

**Итог:** параллельные загрузки в браузере могут **лучше забить** доступную полосу моста, но потолок задаёт **мост + Tor**, не число ядер VPS. Для стабильных звонков важнее **низкий jitter и без рестартов Tor**, чем гонка за 100 Mbit/s через мост.

## Wi‑Fi медленнее LTE при том же `link=tor`

Трафик до VPS один и тот же Tor на сервере. Разница обычно **до VPS**:

- DPI/шейпинг домашнего провайдера;
- роутер (QoS, VPN, слабый Wi‑Fi);
- другой маршрут LTE.

Сравнивайте один инстанс Olcbox, один и тот же тест.
